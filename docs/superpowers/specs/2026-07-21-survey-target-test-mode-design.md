# 조사대상자 기반 테스트 모드 설계

> 작성일: 2026-07-21
> 상태: 사용자 설계 승인 완료, 문서 검토 대기
> 범위: 기존 테스트 모드를 조사대상자·메일·현황까지 포함하는 전역 테스트 작업공간으로 확장

## 1. 배경

현재 테스트 모드는 `?test=<token>` 링크로 공개 응답 플로우를 반복 검증하는 기능이다.
테스트 응답은 `survey_responses.isTest=true`로 저장되어 통계·쿼터·분석·내보내기에서
제외되지만, 조사대상자와 연결되지 않고 메일 템플릿의 테스트 발송 링크도 실제 응답이
불가능한 sandbox 링크다.

운영 전 검증에는 다음 흐름도 필요하다.

1. 가짜 조사대상자를 만든다.
2. 실제 메일 템플릿과 단체발송 기능으로 메일을 보낸다.
3. 대상자별 초대 링크로 설문에 응답한다.
4. 응답 현황·조사 대상·리포트·메일 상태를 실제 운영과 같은 화면에서 확인한다.
5. 테스트 데이터를 보관하거나 일괄 삭제하고 실제 현황으로 돌아간다.

이 기능은 별도 테스트 화면이나 복제 설문을 만들지 않는다. 기존 운영 화면과 테이블에
명시적인 테스트 범위를 추가해 실제 로직을 그대로 검증한다.

## 2. 목표와 성공 기준

### 목표

- 테스트 모드 ON 시 운영 콘솔 전체가 테스트 데이터 범위로 전환된다.
- 실제 설문은 테스트 모드 중에도 중단되지 않으며 실제 응답과 실제 메일 처리는 계속된다.
- 최대 20명의 테스트 조사대상자를 빠르게 만들 수 있다.
- 기존 메일 템플릿과 단체발송 흐름으로 테스트 메일을 실제 받은편지함에 보낼 수 있다.
- 실제 데이터와 테스트 데이터는 조회·수정·삭제 전 경로에서 교차되지 않는다.
- 보관 종료와 전체 초기화의 의미가 명확하다.

### 성공 기준

- 한 관리자가 테스트 모드를 켜면 같은 설문을 보는 모든 관리자에게 전역 상태가 반영된다.
- 테스트 현황에는 테스트 데이터만, 실제 현황에는 실제 데이터만 보인다.
- 테스트 대상자에게 보낸 메일의 대상자별 링크가 실제 응답으로 이어진다.
- 익명 테스트는 기존처럼 반복 응답이 누적된다.
- 대상자 기반 테스트는 대상자별 마지막 응답 1건만 유지한다.
- 테스트 모드를 꺼도 실제 설문 응답 수집과 실제 캠페인 처리는 영향을 받지 않는다.
- 테스트 데이터 초기화가 실제 대상자·실응답·실캠페인·공유 템플릿을 변경하지 않는다.

## 3. 검토한 접근

### A. 기존 테이블에 테스트 범위 추가 — 채택

- `contact_targets`, `survey_responses`, `mail_campaigns`를 `isTest`로 구분한다.
- 기존 화면과 서비스를 재사용하고 데이터 범위만 전환한다.
- 실제 운영 로직과 테스트 로직의 드리프트가 가장 적다.
- 모든 조회·mutation에 범위 가드를 빠짐없이 적용해야 한다.

### B. 테스트 전용 테이블·화면 구축 — 기각

- 격리는 강하지만 조사대상·PII·응답·메일 기능을 거의 복제해야 한다.
- 시간이 지나면서 실제 흐름과 테스트 흐름이 달라질 위험이 크다.

### C. 테스트용 설문 복제 — 기각

- 기존 기능을 그대로 쓸 수 있지만 원본 질문·발행본·템플릿·컬럼 변경을 계속 동기화해야 한다.
- 테스트 결과가 원본 설문의 운영 콘솔과 분리되어 관리가 번거롭다.

## 4. 핵심 제품 규칙

| 항목 | 결정 |
| --- | --- |
| 테스트 상태 | 설문 단위 전역 상태. 한 관리자의 ON/OFF가 모든 관리자에게 적용 |
| 실사 중 ON | 실제 응답·실캠페인은 계속 처리. 운영 콘솔에서만 실제 데이터 숨김 |
| 운영 화면 | 별도 테스트 화면 없이 기존 화면 전체의 데이터 범위 전환 |
| 테스트 컬럼 | 실제 컬럼을 최초 1회 복사한 뒤 `testContactColumns`로 독립 관리 |
| 테스트 대상자 | 자동 생성·수동 추가를 합쳐 최대 20명, 반복 생성은 기존 대상자 뒤에 추가 |
| 테스트 업로드 | 지원하지 않음. 테스트 모드에서 실제 업로드 전 경로 차단 |
| 메일 템플릿 | 실제/테스트 공유 |
| 테스트 캠페인 | 테스트 대상자만 선택, 테스트 이력만 표시, 실제 Resend 발송 |
| 익명 테스트 | 대상자 0명일 때 사용. 같은 브라우저의 반복 응답도 계속 누적 |
| 대상자 테스트 | 대상자별 마지막 응답 1건만 유지. 별도 재시작 UI 없음 |
| 보관 종료 | 데이터 보관, 진행 중 응답 재개 가능, 남은 테스트 메일 발송 취소 |
| 전체 초기화 | 테스트 대상자·응답·메일·컬럼 전체 삭제, 남은 테스트 발송 취소 |
| 내보내기 | 항상 사용 가능. 기존처럼 테스트 응답은 무조건 제외, UI 문구 변경 없음 |
| 메일 비용 | 실제 비용 정확성을 위해 테스트 발송도 정산에 포함, 테스트 배지 표시 |

## 5. 데이터 모델

### 5.1 `surveys`

기존 `testModeEnabled`, `testToken`을 유지하고 다음 컬럼을 추가한다.

| 컬럼 | 타입 | 기본값 | 용도 |
| --- | --- | --- | --- |
| `testContactColumns` | JSONB `ContactColumnScheme` nullable | null | 테스트 조사대상자 전용 컬럼 스킴 |

`testContactColumns`는 발행 스냅샷에 넣지 않는 라이브 운영 설정이다. 실제
`contactColumns`와 독립적이며 실제 업로드나 컬럼 편집으로 자동 변경되지 않는다.

### 5.2 `contact_targets`

| 컬럼 | 타입 | 기본값 | 용도 |
| --- | --- | --- | --- |
| `isTest` | boolean NOT NULL | false | 테스트 조사대상자 여부 |

대상자 번호는 실제/테스트 범위별로 독립 발번한다. 기존 유일성 제약은
`(survey_id, resid)`에서 `(survey_id, is_test, resid)`로 바꾸고,
`next_contact_resid`도 테스트 범위를 인자로 받아 같은 범위만 계산한다.

`inviteToken`, `inviteCode`, `unsubscribeToken`은 기존처럼 전역 유일하다.

### 5.3 `survey_responses`

기존 `isTest` 컬럼을 사용한다. 대상자 기반 테스트의 마지막 응답 1건 규칙을 DB에서도
보강하기 위해 다음 partial unique index를 둔다.

```text
UNIQUE(contact_target_id)
WHERE is_test = true
  AND contact_target_id IS NOT NULL
  AND deleted_at IS NULL
```

익명 테스트 응답은 `contactTargetId=null`이라 이 제약의 영향을 받지 않고 계속 누적된다.

### 5.4 `mail_campaigns`

| 컬럼 | 타입 | 기본값 | 용도 |
| --- | --- | --- | --- |
| `isTest` | boolean NOT NULL | false | 테스트 캠페인 여부 |

캠페인 회차도 범위별로 독립 발번한다. 유일성 제약은
`(survey_id, run_number)`에서 `(survey_id, is_test, run_number)`로 바꾸고,
`next_campaign_run_number`가 범위별 캠페인만 계산하도록 확장한다.

### 5.5 하위 테이블

- `contact_pii`, `contact_attempts`: `contact_targets.isTest`를 상속한다.
- `mail_recipients`: `mail_campaigns.isTest`와 `contact_targets.isTest`가 일치해야 한다.
- `response_answers`, `response_edit_logs`: `survey_responses.isTest`를 상속한다.
- `contact_uploads`: 테스트 업로드가 비범위이므로 변경하지 않는다.
- `mail_templates`: 공유 자원이므로 변경하지 않는다.

## 6. 테스트 데이터 범위

서버 전용 공통 타입과 해석 함수를 둔다.

```ts
type OperationsDataScope = 'real' | 'test';

function testFlagForScope(scope: OperationsDataScope): boolean {
  return scope === 'test';
}
```

RSC 읽기 경로는 `surveys.testModeEnabled`를 읽어 현재 범위를 계산하고, 같은 페이지에서
호출하는 집계 서비스에 명시적으로 전달한다. mutation은 클라이언트가 보낸 scope를
신뢰하지 않고 매번 DB의 전역 상태를 다시 확인한다.

```text
testModeEnabled=false → scope='real' → is_test=false
testModeEnabled=true  → scope='test' → is_test=true
```

범위 전환 대상:

- 응답 현황: 상태 KPI, 일자별 집계, 응답시간, 이탈 퍼널, 페이지 체류
- 응답자 목록과 응답 상세
- 조사 대상 목록·상세·수동 추가·컬럼 설정·결과코드
- 그룹 리포트와 진척률
- 메일 캠페인 후보·목록·상세·상태
- 메일 템플릿의 대상자 샘플 미리보기

범위 전환 예외:

- SPSS·엑셀 내보내기와 analytics는 항상 실제 응답만 사용하고 테스트 응답을 제외한다.
- 메일 비용 정산은 실제 발송 비용을 반영하기 위해 실제·테스트 캠페인을 모두 포함한다.
- quota 집계·판정 정책은 이번 기능에서 변경하지 않는다. 실제 quota 수치가 테스트 현황에
  섞이지 않도록 테스트 범위에서는 quota KPI와 quota 패널을 숨긴다.

## 7. 전역 모드와 관리자 동기화

`testModeEnabled`는 테스트 링크 유효성과 운영 콘솔 데이터 범위를 함께 제어하는 전역 상태다.

- 한 관리자가 ON 하면 모든 관리자의 운영 콘솔이 테스트 범위로 전환된다.
- 실제 공개 설문과 실제 캠페인은 중단되지 않는다.
- 한 관리자가 OFF 하면 모든 관리자에게 실제 범위가 다시 보인다.
- OFF는 테스트 링크를 즉시 무효화한다.

다른 관리자의 변경을 오래된 화면이 놓치지 않도록 헤더 제어 컴포넌트는 window focus와
10초 주기 폴링으로 `control.get`을 다시 확인한다. 상태 변화 감지 시 `router.refresh()`한다.
갱신 전의 오래된 화면에서 발생한 mutation은 서버 전역 상태 가드가 거부한다.

OFF 확인창에는 다음 협업 영향을 안내한다.

> 다른 관리자가 진행 중인 테스트와 발송된 테스트 링크도 중단됩니다.

## 8. 테스트 모드 팝오버

메뉴 이름은 대상자 존재 여부와 무관하게 유지한다.

```text
테스트 링크 복사
테스트 대상자 생성
────────────
테스트 모드 끄기
```

`테스트 링크 복사`의 동작만 대상자 수에 따라 바뀐다.

- 테스트 대상자 0명: 기존 `/survey/{identifier}?test={testToken}` 익명 링크를 복사한다.
- 테스트 대상자 1명 이상: `resid ASC` 첫 번째 테스트 대상자의 `/i/{inviteCode}` 링크를 복사한다.
- 첫 번째 대상자를 삭제하면 다음 `resid` 대상자의 링크를 복사한다.
- 보관 종료 후 다시 ON 하면 같은 첫 번째 대상자 링크를 복사한다.
- 전체 초기화 후 다시 ON 하면 대상자가 없으므로 익명 링크로 돌아간다.

대상자별 나머지 링크는 조사 대상 목록의 기존 Web 링크와 단체메일을 통해 사용한다.
링크 목록 모달이나 별도의 `테스트 대상자 보기`·`테스트 대상자 초대` 기능은 만들지 않는다.

## 9. 테스트 대상자 생성

### 9.1 모달

모달은 한 화면에서 두 값만 입력받는다.

- 생성 인원: 1명 이상, 전체 테스트 대상자 최대 20명
- 메일 받을 테스트 주소: 실제 수신 가능한 이메일 1개

반복 실행은 기존 테스트 대상자 뒤에 추가한다. 현재 17명이면 최대 3명만 더 만들 수 있다.

### 9.2 테스트 컬럼 초기화

첫 생성 시 `testContactColumns`가 null이면 다음 순서로 만든다.

1. 실제 `contactColumns`가 있으면 구조만 깊은 복사한다.
2. 실제 스킴이 없으면 빈 스킴에서 시작한다.
3. 이름·회사·전화번호·이메일 의미의 컬럼을 PII 타입·key·label로 감지한다.
4. 빠진 의미 컬럼만 테스트 전용 기본 컬럼으로 보충한다.

그 뒤 `testContactColumns`는 실제 스킴과 독립적으로 편집·보관된다. 실제 업로드와 실제
컬럼 변경은 테스트 스킴에 영향을 주지 않는다. 전체 초기화만 테스트 스킴을 null로 되돌린다.

### 9.3 가짜 데이터

`src/lib/contacts/test-contact-fixtures.ts`에 명백한 합성 데이터 20세트를 둔다.

- 이름: 테스트 담당자 01~20 같은 합성 이름
- 회사: 테스트기업 01~20 같은 합성 회사명
- 전화번호: 실제 연락 가능성이 없는 테스트 번호
- 이메일: 모달에서 받은 동일한 테스트 주소

이름·전화·이메일은 기존 PII 암호화 경로로 `contact_pii`에 저장한다. 회사는 attrs에 저장한다.
복사된 나머지 컬럼은 빈 값으로 두고 조사 대상 상세·수정 화면에서 편집할 수 있게 한다.

### 9.4 원자성

생성 서비스는 하나의 트랜잭션과 설문 단위 잠금 안에서 다음을 수행한다.

1. 테스트 모드가 아직 ON인지 검증한다.
2. 현재 테스트 대상자 수와 20명 제한을 검증한다.
3. 첫 테스트 대상자 생성이면 기존 익명 테스트 응답을 하드 삭제한다.
4. 필요하면 `testContactColumns`를 초기화한다.
5. 테스트 범위 resid를 발번하고 대상자·PII를 삽입한다.

동시 관리자 요청도 20명 제한과 resid 유일성을 깨지 못해야 한다.

## 10. 업로드 차단

테스트용 엑셀 업로드는 만들지 않는다. 테스트 모드에서는 실제 명단 업로드를 다음 세
경계에서 모두 막는다.

1. UI: 상단 `조사 대상 업로드`, 목록의 `엑셀 업로드`·`+ 업로드`, 이력의 `새 업로드` 비활성화
2. 라우트: `/contacts/upload/new` 직접 접근 시 업로더 대신 안내 카드 표시
3. 서비스: 이미 열린 페이지나 직접 RPC 호출의 `ingestContactUpload` 거부

호버와 키보드 focus 안내:

> 테스트 모드에서는 실제 조사대상자를 업로드할 수 없습니다. 테스트 대상자는 상단의
> ‘테스트 대상자 생성’을 이용하세요.

`+ 조사 대상 추가`는 유지하며 테스트 모드에서 `isTest=true` 대상자를 수동 추가한다.
수동 추가도 전체 20명 제한에 포함한다.

## 11. 익명 테스트와 대상자 테스트

### 11.1 익명 테스트

테스트 대상자가 0명일 때 기존 테스트 링크를 사용한다.

- 같은 브라우저·기기에서 반복해도 매번 새 테스트 응답으로 누적한다.
- 중복 감지를 우회한다.
- `contactTargetId`는 null이다.

첫 테스트 대상자를 만들면 기존 익명 테스트 응답을 삭제한다. 이전에 복사한 익명 링크가
대상자 테스트와 병행되지 않도록, 서버는 테스트 대상자가 한 명이라도 있을 때 익명
`?test` 링크의 신규 진입·저장을 거부한다. 마지막 테스트 대상자가 삭제되면 익명 링크가
다시 유효해진다.

### 11.2 대상자 테스트 링크

`/i/{inviteCode}` 해석 결과에 `contact_targets.isTest`를 포함한다.

- 실제 대상자 링크: 기존 실제 응답 정책 적용
- 테스트 대상자 링크: `testModeEnabled=true`일 때만 테스트 세션으로 허용
- 실제 대상자 ID와 테스트 토큰, 테스트 대상자 ID와 실제 응답 범위의 혼합은 거부
- OFF 상태에서는 신규 진입·재개·저장 모두 거부

테스트 링크 유효성은 서버가 초대 코드와 설문 전역 상태로 판단한다. 클라이언트가 보낸
`isTest` boolean은 신뢰하지 않는다.

### 11.3 대상자별 마지막 응답 1건

테스트 대상자 한 명은 활성 테스트 응답 한 건만 가진다.

- 기존 응답이 `in_progress`면 링크 재진입 시 그대로 재개한다.
- 기존 응답이 완료·중단 같은 terminal 상태면 링크 재진입 시 별도 확인 UI 없이 같은 응답을
  새 테스트로 초기화한다.
- 초기화 시 질문 응답 JSON, `response_answers`, 수정 로그, 완료·활동·진행률·페이지 방문·시간
  정보를 모두 새 응답 상태로 되돌린다.
- 응답 현황과 응답자 목록에는 항상 대상자별 마지막 한 건만 존재한다.
- `다시 테스트하기` 버튼이나 응답 누적 이력은 만들지 않는다.

## 12. 메일

### 12.1 템플릿과 후보

- 메일 템플릿은 실제/테스트가 공유한다.
- 테스트 모드의 템플릿 미리보기 샘플은 테스트 대상자에서 가져온다.
- 단체발송 후보는 `contact_targets.isTest=true` 대상자만 표시한다.
- 기존 필터·전체 선택·preflight·캠페인 작성 UI를 그대로 사용한다.
- 별도 초대 화면과 대상자 생성 직후 자동발송은 만들지 않는다.
- 템플릿 편집 화면의 기존 단독 `테스트 발송`은 계속 sandbox 링크를 사용한다. 실제로
  응답 가능한 대상자별 링크 검증은 테스트 캠페인 발송에서만 수행한다.

### 12.2 캠페인 생성

서버가 현재 전역 모드를 다시 읽고 `mail_campaigns.isTest`를 결정한다. 선택된 모든 대상자의
`isTest`가 캠페인 범위와 일치해야 한다. 테스트 캠페인 작성 중 다른 관리자가 모드를 끄면
실캠페인으로 강등하지 않고 제출을 거부해 화면 새로고침을 요구한다.

테스트 캠페인:

- 제목 앞에 `[TEST]` 자동 추가
- 본문 하단에 테스트 메일 안내 푸터 표시
- 대상자별 `/i/{inviteCode}` 링크는 실제 응답 가능
- 수신거부 링크만 sandbox로 치환해 어떤 대상자 데이터도 변경하지 않음
- 발송·도착·열람·반송·실패 상태를 일반 캠페인과 동일하게 기록
- 실제 Resend 비용이 발생하므로 메일 비용 정산에 포함하고 `테스트` 배지 표시

테스트 모드 중에도 이미 진행 중인 실제 캠페인은 계속 처리된다.

### 12.3 OFF와 발송 취소

`보관하고 끄기`와 `삭제 후 끄기` 모두 아직 보내지 않은 테스트 메일 발송을 취소한다.
dispatcher는 prepare뿐 아니라 각 chunk 시작 전에도 캠페인 상태를 검사해 취소 뒤 추가 발송을
막는다. 이미 Resend에 전달된 메일은 회수할 수 없다.

삭제 뒤 들어오는 webhook과 이미 시작한 백그라운드 step은 대상 캠페인·수신자 행이 없으면
성공적인 no-op으로 종료해야 한다.

## 14. 종료 다이얼로그와 수명주기

현재 3버튼 다이얼로그의 디자인과 구조를 유지한다.

테스트 대상자가 없을 때 제목:

> 테스트 응답 N건을 삭제할까요?

테스트 대상자가 있을 때 제목:

> 테스트 대상자 N명과 응답 M건을 삭제할까요?

버튼:

- `취소`: 아무 작업도 하지 않는다.
- `보관하고 끄기`: 테스트 데이터는 유지하고 모드만 OFF한다.
- `삭제 후 끄기`: 테스트 데이터 전체를 삭제하고 모드를 OFF한다.

### 14.1 보관하고 끄기

1. 테스트 모드를 OFF해 모든 테스트 링크와 저장을 차단한다.
2. queued·sending 테스트 캠페인을 취소하고 아직 시작하지 않은 발송을 중단한다.
3. 대상자·응답·메일 이력·테스트 컬럼을 유지한다.
4. `in_progress` 응답 상태도 그대로 유지한다.
5. 다시 ON 하면 진행 중 응답을 재개하고 첫 테스트 대상자 링크를 다시 복사할 수 있다.

### 14.2 삭제 후 끄기

1. 테스트 모드를 OFF한다.
2. 테스트 캠페인을 취소한다.
3. 테스트 응답과 하위 답변·로그를 삭제한다.
4. 테스트 메일 수신자와 캠페인을 삭제한다.
5. 테스트 대상자와 PII·attempt를 삭제한다.
6. `testContactColumns`를 null로 되돌린다.

확인 설명:

> 테스트 데이터를 모두 삭제하고 진행 중인 발송을 중단합니다. 이미 발송된 메일은 취소할 수
> 없습니다.

열린 테스트 응답 화면은 다음 요청에서 테스트 모드 종료 또는 데이터 초기화 안내를 표시하고
로컬 세션을 정리한다.

## 15. 개별 테스트 대상자 삭제

테스트 대상자 개별 삭제는 같은 대상자의 마지막 테스트 응답과 메일 수신자 행을 함께 제거한다.
수신자 삭제로 영향을 받은 테스트 캠페인의 카운터는 트랜잭션 안에서 재계산한다.

마지막 테스트 대상자가 삭제되면:

- 대상자 기반 테스트가 종료된다.
- 기존 익명 `?test` 링크가 다시 유효해진다.
- `테스트 링크 복사`는 익명 링크를 복사한다.
- `testContactColumns`는 보관하며 전체 초기화 때만 제거한다.

실제 대상자·실응답·실캠페인은 변경하지 않는다.

## 16. 오류 처리와 경쟁 조건

다음 요청은 서버에서 거부한다.

- OFF 상태의 테스트 대상자 생성·수정·메일 발송
- ON 상태의 실제 조사대상자 업로드
- 실제/테스트 대상자 ID 혼합 캠페인
- 반대 범위의 대상자·응답·캠페인 상세 접근
- 테스트 대상자 20명 초과 생성
- 대상자가 존재하는 동안의 익명 테스트 링크 사용
- OFF 상태의 테스트 링크 신규 진입·재개·저장

교차 범위 ID는 권한 오류 대신 `NOT_FOUND`와 같은 의미로 처리한다. 모드가 다른 관리자에 의해
변경된 stale 화면의 mutation은 실데이터 동작으로 폴백하지 않고 현재 상태를 다시 불러오도록
안내한다.

초기화와 생성은 설문 단위 잠금과 트랜잭션으로 처리한다. 외부 발송은 DB 트랜잭션으로 되돌릴
수 없으므로 이미 발송된 메일의 비가역성은 UI에서 명시한다.

### 17.1 단위 테스트

- 종료 다이얼로그 제목과 카운트

### 17.2 서비스·통합 테스트

- 테스트 모드 ON 중 실응답·실캠페인 처리가 계속되는지
- 운영 집계·profiles·contacts·report·campaigns의 범위 격리
- 첫 대상자 생성의 익명 응답 삭제·스킴 복사·PII 저장 원자성
- 동시 생성의 20명 제한과 범위별 resid 발번
- 익명 테스트 반복 응답 누적
- 대상자 테스트 `in_progress` 재개와 terminal 응답 자동 초기화
- 대상자별 활성 응답 partial unique 제약
- 실제/테스트 초대 링크 교차 방어
- 테스트 후보만으로 캠페인 생성, stale 모드 제출 거부
- 보관 종료·전체 초기화·개별 삭제와 캠페인 카운터 정합성
- 취소 이후 dispatcher chunk 중단
- 삭제 후 webhook·Inngest step no-op
- 업로드 UI 우회 RPC 차단
- SPSS·엑셀 내보내기 테스트 응답 제외
- 메일 비용에 테스트 발송 포함

### 17.3 실DB 테스트

- `(survey_id, is_test, resid)` 유일성
- `(survey_id, is_test, run_number)` 유일성
- 테스트 대상자별 활성 응답 partial unique index
- 범위별 `next_contact_resid`, `next_campaign_run_number`
- 삭제 순서와 FK cascade·set null 동작

### 17.4 수동 검증

1. 두 관리자 브라우저에서 전역 ON/OFF 동기화 확인
2. 익명 테스트를 같은 브라우저에서 반복해 응답 누적 확인
3. 테스트 대상자 생성 후 익명 응답 삭제와 링크 전환 확인
4. 같은 받은편지함으로 여러 개인화 메일과 서로 다른 초대 링크 수신
5. 대상자 완료 후 같은 링크 재진입 시 별도 UI 없이 마지막 응답 교체
6. 보관 종료 후 링크 차단, 재ON 후 진행 응답 재개
7. 전체 초기화 후 실제 데이터 보존과 익명 링크 복귀
8. 실제 캠페인이 테스트 모드 중에도 계속 처리되는지 확인

- 테스트 대상자 생성 직후 자동발송
- 테스트 링크 목록 모달
- `테스트 대상자 보기`, `테스트 대상자 초대`, `다시 테스트하기` 전용 UI
- 테스트 응답 SPSS·엑셀 내보내기
- 테스트 대상자 총 20명 초과 생성
- 실제 컬럼 변경을 기존 `testContactColumns`에 자동 재동기화
- 테스트 quota 시뮬레이션 또는 기존 quota 판정 정책 변경

## 19. 마이그레이션과 기존 데이터

- 기존 `contact_targets`, `mail_campaigns` 행은 `isTest=false`로 backfill한다.
- 기존 `survey_responses.isTest=true` 행은 익명 테스트 응답으로 유지한다.
- `testContactColumns`는 null로 시작한다.
- 기존 테스트 토큰과 익명 테스트 링크 정책은 대상자가 없을 때 그대로 유지한다.
- 커스텀 SQL 함수와 유일성 제약 변경은 `supabase/migrations`에 명시적으로 반영하고
  `manual-migrations.json` 및 CI migration gate 규칙을 따른다.

## 20. 구현 단계 경계

이 문서는 하나의 기능 설계지만 구현 계획은 다음 의존 순서를 따라야 한다.

1. 스키마·마이그레이션·범위 helper
2. 읽기 쿼리 범위 전환과 전역 배너
3. 테스트 컬럼·대상자 생성·수동 CRUD
4. 익명/대상자 링크 전환과 마지막 응답 1건 수명주기
5. 메일 후보·캠페인·발송 안전장치
6. 보관 종료·전체 초기화·개별 삭제
7. 전역 관리자 동기화와 전체 회귀 검증

각 단계는 실제 데이터 경로가 계속 `isTest=false`로 고정되는 회귀 테스트를 포함해야 한다.
